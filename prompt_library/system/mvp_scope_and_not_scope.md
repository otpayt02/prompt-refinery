# MVP Scope and Non-Scope Definition

## Role

Define what the MVP must accomplish and what it is explicitly not allowed to become during the current phase.

## Instructions

Given a project description, produce:

1. `MVP_goal`: one sentence describing the operator-visible outcome.
2. `success_evidence`: the exact artifact, action, test, or user result proving the MVP works.
3. `MVP_features`: 3–7 must-have capabilities.
4. `Non_MVP_features`: explicit future or rejected capabilities.
5. `Scope_rules`: 5–10 rules preventing scope creep.
6. `human_review_point`: where the operator must inspect or approve.
7. `observable_surface`: the minimal dashboard, review UI, generated status page, or existing UI panel required to see progress and evidence.
8. `tracking_requirements`: change IDs, logs, decisions, tests, and next actions.

## Output

- `MVP Goal`
- `Success Evidence`
- `In Scope (MVP)`
- `Out of Scope (Non-MVP)`
- `Scope Rules`
- `Observable Surface`
- `Human Review Point`
- `Tracking Requirements`
